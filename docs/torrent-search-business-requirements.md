# Бизнес-требования — Интеллектуальный механизм поиска торрент-релизов

## Наименование
Разработка интеллектуального механизма поиска торрент-релизов.

## Бизнес-потребность
Повысить полноту, скорость и качество поиска фильмов и сериалов путем реализации интеллектуального механизма поиска, использующего несколько поисковых запросов, несколько торрент-индексаторов, автоматическую обработку результатов и их ранжирование.

## Цель
Обеспечить пользователю максимально полный список релевантных торрент-релизов независимо от особенностей оформления названия на различных торрент-трекерах.

## Стейкхолдеры

- Пользователь приложения
- Владельцы мультимедийной библиотеки
- Разработчики приложения

## Проблематика (AS-IS)
В текущей реализации поиск выполняется по одному поисковому запросу.

Особенности существующей реализации:

- поиск осуществляется только по одной поисковой строке;
- различия в названиях релизов на разных торрент-трекерах приводят к потере части результатов;
- результаты поиска содержат дублирующиеся релизы;
- сортировка выполняется только по количеству сидов;
- отсутствует интеллектуальное ранжирование результатов;
- отсутствует возможность поиска по альтернативным названиям фильма или сериала;
- отсутствует разделение поиска фильмов, сериалов и аниме;
- результаты отображаются только после завершения поиска по всем индексаторам.
В результате пользователь получает неполную выдачу и может не увидеть наиболее качественный релиз.

## TO-BE
Разработать интеллектуальный поисковый механизм, выполняющий:

- генерацию нескольких поисковых запросов;
- параллельный поиск по нескольким торрент-индексаторам;
- объединение результатов;
- удаление дублирующихся релизов;
- интеллектуальную сортировку результатов;
- фильтрацию нерелевантных релизов;
- возможность потокового отображения найденных результатов.

## Функциональные требования

### ФТ-01. Генерация поисковых запросов
Система должна автоматически формировать несколько вариантов поисковых запросов на основании данных о фильме или сериале.

При генерации должны использоваться:

- локализованное название;
- оригинальное название;
- год выпуска;
- альтернативные названия (при наличии).
Например:

```
История игрушек 5 2026
Toy Story 5 2026

История игрушек 5
Toy Story 5

История игрушек
Toy Story
```

### ФТ-02. Поиск по нескольким запросам
Система должна выполнять поиск по каждому сформированному варианту поискового запроса.

Все найденные результаты должны объединяться в единую коллекцию.

### ФТ-03. Параллельный поиск
Система должна выполнять поиск одновременно по нескольким торрент-индексаторам.

Поиск по одному индексатору не должен блокировать выполнение поиска по другим индексаторам.

Ошибка или недоступность отдельного индексатора не должна приводить к завершению поиска с ошибкой.

### ФТ-04. Настройка используемых индексаторов
Пользователь должен иметь возможность включать и отключать используемые торрент-индексаторы.

Поиск должен выполняться только по выбранным пользователем индексаторам.

### ФТ-05. Разделение индексаторов по типу контента
Система должна учитывать тип искомого контента.

При поиске фильмов должны использоваться только индексаторы, предназначенные для фильмов.

При поиске сериалов должны использоваться индексаторы, содержащие сериалы.

При поиске аниме должны использоваться специализированные аниме-индексаторы.

### ФТ-06. Ограничение времени поиска
Для каждого индексатора должно использоваться ограничение времени ожидания ответа.

По истечении установленного времени поиск по конкретному индексатору должен прекращаться без влияния на остальные поисковые запросы.

### ФТ-07. Объединение результатов
После завершения поиска система должна объединить результаты, полученные от всех индексаторов и всех поисковых запросов.

### ФТ-08. Удаление дубликатов
Система должна автоматически определять дублирующиеся релизы.

При обнаружении нескольких одинаковых релизов пользователю должна отображаться только одна запись.

Определение дубликатов должно выполняться по следующим признакам:

- InfoHash;
- нормализованное название;
- размер файла;
- год выпуска.

### ФТ-09. Нормализация данных
Перед обработкой результатов система должна привести данные всех индексаторов к единой модели.

Для каждого релиза должны быть определены:

- название;
- качество;
- год выпуска;
- размер;
- количество сидов;
- количество пиров;
- тип озвучки;
- источник;
- сезон;
- серия;
- трекер;
- magnet-ссылка.

### ФТ-10. Интеллектуальное ранжирование
Система должна рассчитывать рейтинг каждого найденного релиза.

При расчете рейтинга должны учитываться:

- совпадение названия;
- совпадение года выпуска;
- совпадение сезона;
- совпадение серии;
- качество видео;
- тип озвучки;
- количество сидов;
- приоритет торрент-индексатора.
Пользователю должны отображаться релизы с наибольшим рейтингом.

### ФТ-11. Фильтрация нерелевантных результатов
Система должна автоматически исключать результаты, не относящиеся к искомому контенту.

Например:

- другие части фильма;
- сборники;
- трейлеры;
- саундтреки;
- дополнительные материалы.

### ФТ-12. Потоковое отображение результатов
При получении результатов поиска система должна отображать найденные релизы без ожидания завершения поиска по всем индексаторам.

По мере поступления новых результатов список должен автоматически обновляться.

### ФТ-13. Кэширование результатов поиска
Система должна сохранять результаты поиска на ограниченный период времени.

При повторном поиске идентичного запроса система должна использовать сохраненные результаты без повторного обращения к торрент-индексаторам.

### ФТ-14. Статистика работы индексаторов
Система должна вести статистику работы каждого торрент-индексатора.

Должны фиксироваться:

- среднее время ответа;
- количество успешных запросов;
- количество ошибок;
- количество таймаутов;
- дата последнего успешного ответа.

### ФТ-15. Автоматическое исключение недоступных индексаторов
При многократном возникновении ошибок система должна временно исключать проблемный индексатор из поиска.

После истечения установленного периода времени система должна автоматически повторять попытку использования индексатора.

## Нефункциональные требования

- Поиск должен выполняться параллельно.
- Ошибка одного индексатора не должна приводить к ошибке поиска.
- Архитектура должна обеспечивать возможность подключения новых торрент-индексаторов без изменения логики поиска.
- Архитектура должна обеспечивать возможность подключения альтернативных поисковых сервисов (Torznab, Prowlarr и др.).
- Логика генерации поисковых запросов, объединения результатов, удаления дубликатов и ранжирования должна быть реализована в виде независимых модулей.
- Архитектура должна обеспечивать возможность масштабирования количества одновременно используемых торрент-индексаторов.
- Настройки используемых индексаторов должны храниться отдельно от программного кода и изменяться пользователем без необходимости модификации приложения.
